# Design: EverMCP — MCP tool orchestration framework

Generated by /office-hours on 2026-06-24
Last revised: 2026-06-25 (v0.2.0 — stripped bundled tools)
Status: APPROVED
Mode: Builder

## Problem Statement

家里/工作室有多台闲置设备（笔记本、台式机、嵌入式开发板等），每台都有不同算力（GPU、NPU、CPU）但当前没有被充分利用。这些设备的能力应该以 MCP（Model Context Protocol）暴露给 AI Agent，让 AI 能调用本地工具完成真实任务。

**v1.0 范围**：本地单进程 MCP server，**不 ship 任何工具**。框架 + 安全模型 + stdio transport + 文档 + examples/tools/ 下 2 个参考工具。架构按"协调器 + worker"分层，为 v2 跨设备调度预留接口。

**v2 范围**（不实现）：跨设备任务调度，AI 一次请求，调度器自动选最合适的设备执行。

## What Makes This Cool

- **多设备扇出**（v2）：家里所有设备变成 AI Agent 的"工具车间"。让 AI 转码一个 4K 视频，调度器自动选台式机 GPU 来跑，AI 完全不知道也不需要知道是哪台机器。
- **Agent 自进化**（v3+）：未来 Agent 在运行时能自己写新工具并注册到调度器。架构里通过 `tools/` hot-reload 留了 hook。
- **闲置算力变服务**：把家里零散的硬件（消费级 GPU、嵌入式 NPU、老笔记本）变成 AI 工具的"基础设施"。
- **工具来源解耦**（v0.2.0+）：项目本身不带工具，用户通过 `--tools-dir` 指向自己的工具目录。框架 + 安全 + transport 是核心；工具生态留给用户。

v1 不实现多设备和 Agent 自进化，但**架构必须支持**，否则 v2/v3 要重写。

## Constraints

- 单人开发
- 自定节奏（builder 模式）
- v1 目标：本地单设备 + 框架 + 文档 + 2 个示例工具；端到端跑通
- 部署目标：v1 是开发者在自己 Python 环境跑；v2 才会涉及消费级硬件（无需 Python 环境）
- v1 唯一 AI 客户端目标：**Claude Desktop**（Cursor 用户 README 里有迁移提示，v1 不做官方支持）
- v1 MCP transport: stdio
- **v1.0 项目不 ship 任何工具**：tools/ 目录在 git 里空（.gitkeep）；examples/tools/ 提供参考

## Premises

**P1. 不从自定义框架开始** — 用现成 MCP 框架（官方 Python SDK）起手，只在它不能解决的地方（设备注册 + 路由）造东西。

**P2. v1 是单设备 + 多工具，不是多设备 + 少工具** — 多设备的复杂度（发现、注册、心跳、鉴权、负载均衡）在只有一个工具时全是浪费。

**P3. 框架不带工具是更干净的边界** — 工具是用户域知识，不该被项目强加。examples/ 提供可复制模板 + 文档说怎么写。仓库小、依赖少、争议少。

**P4. MCP 是对的协议，但留 protocol adapter 抽象** — v1 锁 MCP；长期生态可能分裂，保留抽象层。

**P5. Agent 自进化（v3+ 愿景）留 hook，v1 不实现** — 架构里给"Agent 写新工具"留位置（`tools/` 目录 hot-reload、任何符合规范的脚本自动注册），v1 不做"Agent 自己写"的逻辑。

## Approaches Considered

### Approach A: 极简单进程 v1

- **Summary**: 一个 Python 进程，官方 MCP SDK 暴露工具，FFmpeg 直接是 Python 函数。`tools/` 目录扫描，每个 `.py` 一个工具。stdio 模式跟 AI 客户端通信。完全不考虑多设备。
- **Effort**: S（1 周）
- **Risk**: Low
- **Pros**: 一个周末能跑，代码 < 300 行；立即给 AI 用；验证 FFmpeg 集成范式
- **Cons**: 多设备时几乎要重写；没有热加载基础；没有协议适配层
- **Reuses**: 官方 MCP Python SDK, FFmpeg subprocess

### Approach B: 插件化 v1 + 多设备预留（推荐）

- **Summary**: 同样单进程 v1，但内部按"协调器 + worker"形态设计。Worker 跑在进程内（假装是另一台设备），工具通过 worker 协议调用。后期拆成独立进程只换传输层，核心逻辑不变。`tools/` 目录是 worker 物理位置，热加载。MCP 层有 protocol adapter 接口。
- **Effort**: M（3-4 周全职 / 6-8 周业余）
- **Risk**: Med
- **Pros**: v1 仍是单设备+多工具（简单）；架构对多设备友好；worker 协议可在进程内快速测试；自然满足 P1/P4/P5
- **Cons**: 比 A 多 1-2 周；内部抽象多一些；如果永远不做多设备 = 过度设计
- **Reuses**: 官方 MCP SDK，in-process function call (v2 拆进程时切 gRPC over TCP)

### Approach C: 纯 P2P 设备即服务（野路子）

- **Summary**: 跳过"中心协调器"概念，每台设备跑同一二进制，既是 MCP server 又是 worker。通过 mDNS 自动发现其他设备。AI 客户端连任何一台就能调全网工具。设备间通过加密 libp2p 通道直连。
- **Effort**: XL（2+ 月）
- **Risk**: High
- **Pros**: 零中心、零单点；设备即插即用；极客感最强
- **Cons**: 实现复杂度 L+；鉴权/加密要从零设计；NAT/防火墙问题；嵌入式 NPU 设备可能跑不动
- **Reuses**: libp2p, mDNS, mcp-framework, Noise protocol

### Approach D: 最小框架 + 用户自带工具（v0.2.0 采纳）

- **Summary**: 跟 B 同架构，但 v1.0 **不 ship 任何工具**。`tools/` 目录在 git 里空（.gitkeep），`examples/tools/` 提供 2 个参考模板（demo.hello + io.read_file）。用户通过 `--tools-dir` 指向自己的工具目录。
- **Pros**: 仓库小、依赖少；不强加用户不需要的工具；examples 够学习；v2 路径不变
- **Cons**: 第一次跑没示例工具（用户需 `--tools-dir examples/tools` 看到 demo）；文档要求高（用户得看 docs/adding-tools.md 才能上手）
- **Reuses**: B 的全部 + 新增 CLI `--tools-dir` flag + `EVERMCP_TOOLS_DIR` env var

## Recommended Approach

**D: 最小框架 + 用户自带工具**（v0.2.0 起取代 B）

B 解决了"v1 单设备多工具但架构对多设备友好"。D 是 B 的进一步收紧：v1.0 连示例工具都不带，工具目录完全用户控制。A 太单薄（多设备愿景不值），C 太野（v1 不该这么重）。

### 决策记录

下面是关键技术选型，已经决定（v1.0 不再讨论）：

- **语言**: **Python**（FFmpeg 生态、subprocess 简单、NPU 工具链一般 Python）
- **MCP SDK**: **官方 Python MCP SDK**（不是 mcp-framework.com — 后者是 TypeScript）
- **MCP transport**: **stdio**（Claude Desktop 原生支持；v2 加 SSE）
- **进程内 worker 协议**: **v1 直接函数调用**（Coordinator 直接调 LocalWorker 方法，dict 在内存传，无序列化无 dispatcher）。v2 拆进程时**新增** JSON-RPC over gRPC 序列化层（v1 没这层，v2 才有；不要让"协议层零改动"的执念把 v1 复杂化）
- **工具 hot-reload**: **`watchdog`** 文件系统 watcher（实时，资源占用低）
- **FFmpeg 进程模型**: **同步执行 + 超时 + `-progress` flag**（v1 简单；v2 异步 + 进度推送）
- **配置**: **TOML 文件 + env var override**（标准做法）
- **工具目录**: **`--tools-dir` CLI flag** + **`EVERMCP_TOOLS_DIR` env var** + 默认 = `<repo>/tools`（git 里空）；examples/tools/ 是参考实现

### 核心架构图

```
┌─────────────────────────────────────┐
│ Coordinator (协调器)                │
│  - MCP server (v1 = stdio, MCP SDK) │
│  - 工具注册表 (Tool Registry)        │
│  - Worker 派发器 (Worker Dispatcher) │
└──────────────┬──────────────────────┘
               │ 进程内直接函数调用 (v1, dict 内存传递)
               │ 跨网络 gRPC + JSON-RPC 序列化 (v2)
┌──────────────▼──────────────────────┐
│ LocalWorker (本地 worker)            │
│  - 工具执行环境                       │
│  - 设备能力描述 (见 Capability Schema)│
│  - tools/ 目录 hot-reload            │
└─────────────────────────────────────┘
```

**注**: v1 是单 worker 场景，"Worker Dispatcher" 当前是单点派发（v2 才需要真实路由逻辑）。`MCP server` 就是 v1 的唯一协议入口（v2 才抽出 Protocol Adapter 概念）。

**v1.0 状态**:
- 协调器和 worker 在同一进程
- worker 协议是 **in-process function call**（无序列化、无 dispatcher；JSON-RPC 序列化层 v2 才加）
- 没有真实网络层
- `tools/` 目录在 git 里**空**（`.gitkeep`），hot-reload 工作（用户填自己的工具时）
- `examples/tools/` 提供 2 个参考实现（demo.hello + io.read_file）
- 用户通过 `evermcp serve --tools-dir <path>` 加载自己的工具
- MCP 暴露 N 个工具（N 由 `--tools-dir` 决定，0 到 N）给 AI 客户端

**v2 演进路径**（不实现，但架构支持）:
- worker 拆成独立进程
- 协调器和 worker 用 gRPC
- 加 mDNS 设备发现
- 加 capability-based 路由

### Tool Authoring Contract

工具作者必须遵守的契约。这是 v1 最重要的一个约定。

**命名空间规则**: 工具全名 = `<category>.<function_name>`，其中 `category` 来自**文件路径**（不是装饰器参数），`function_name` 来自函数名。

- `tools/media/transcode.py` 的 `transcode()` → `media.transcode`
- `tools/io/read_file.py` 的 `read_file()` → `io.read_file`
- `tools/http/get.py` 的 `get()` → `http.get`

**支持的类型注解**（自动转 JSON Schema）:
- 基本: `str`, `int`, `float`, `bool`
- 容器: `list[T]`, `dict[str, T]`
- 复合: `Optional[T]`, `Literal["a", "b"]`
- 范围/约束: 用 Pydantic `Field(..., ge=0, le=51)` 描述

**函数签名**:
```python
from typing import Literal
from pydantic import Field
from typing_extensions import Annotated
from evermcp.core.tool import tool, ToolContext

# 文件: examples/tools/demo/hello.py

@tool(
    description="Say hello to someone by name.",
)
def hello(
    name: Annotated[str, Field(min_length=1, max_length=100,
                               description="Person's name.")],
    ctx: ToolContext | None = None,  # 可选，注入 cwd/logger/safe_path/safe_url/config
) -> dict:
    """Greet someone by name."""
    return {"message": f"hello, {name}"}

# 工具全名: demo.hello
```

**契约规则**:
1. 工具文件路径: `<tools_dir>/<category>/<name>.py`（`<tools_dir>` 由 `--tools-dir` 或 env 决定）
2. 工具全名: `<category>.<function_name>`，AI 客户端按这个调用
3. 每个工具函数用 `@tool(description=...)` 装饰器标注
4. 类型注解 + Pydantic Field 自动转 JSON Schema 暴露给 AI
5. `ctx: ToolContext` 参数可选，提供 `cwd`、`logger`、`safe_path`、`safe_url`、`config`
6. 返回值必须 JSON 可序列化（dict/list/str/int/float/bool/None）
7. 异常被 worker 捕获，转成错误信封返回（见 Error Model）
8. 一个文件可以包含多个 `@tool` 函数

完整 spec 见 [`docs/adding-tools.md`](docs/adding-tools.md)。

### Worker Protocol

**Transport (v1)**: in-process function call. v2: gRPC over TCP.
**Encoding**: JSON.
**Methods**:
- `list_tools() -> list[ToolDescriptor]`
- `call_tool(name: str, args: dict, call_id: str) -> ToolResult`
- `get_capabilities() -> CapabilityDescriptor`

**Schema 定义**:

```python
class ToolDescriptor(TypedDict):
    name: str           # e.g. "media.transcode"
    description: str    # 来自 @tool 装饰器
    input_schema: dict  # JSON Schema (从签名推导)
    category: str       # 来自目录

class ToolResult(TypedDict):
    success: bool
    result: dict | None
    error: ToolError | None

class ToolError(TypedDict):
    code: int           # -32001..-32005
    message: str
    data: dict          # 包含 tool name, args, stderr, traceback(debug)

class CapabilityDescriptor(TypedDict):
    cpu_cores: int
    memory_total_mb: int
    memory_free_mb: int
    disk_free_gb: float
    ffmpeg_encoders: list[str]   # e.g. ["libx264", "libx265", "h264_nvenc", "h264_videotoolbox"]
    gpu_available: bool
    npu_available: bool
    platform: Literal["linux", "macos", "windows"]
```

**完整调用示例**:
```json
// Request
{
  "method": "call_tool",
  "params": {
    "name": "media.transcode",
    "call_id": "uuid-v4-here",
    "args": {
      "input_path": "/data/in.mp4",
      "output_path": "/data/out.mp4",
      "output_format": "mp4",
      "crf": 23
    }
  }
}

// Response (success)
{
  "result": {
    "call_id": "uuid-v4-here",
    "output_path": "/data/out.mp4",
    "duration_s": 120.5,
    "size_bytes": 12345678
  }
}

// Response (error)
{
  "result": null,
  "error": {
    "call_id": "uuid-v4-here",
    "code": -32003,
    "message": "FFmpeg failed with exit code 1",
    "data": {
      "tool": "media.transcode",
      "stderr": "ffmpeg version 6.0 ... Error: Invalid data found ..."
    }
  }
}
```

**幂等性**: `call_id` 由协调器生成（per AI 客户端调用一次）。AI 客户端重试时协调器会生成新 `call_id`，工具会**重新执行**（v1 不做去重）。FFmpeg 这种工具调用时 AI 客户端必须自己保证不重试。

### Security Model

**信任边界**:
- AI 客户端 ↔ 协调器: 不可信（AI 输入当作用户输入处理）
- 协调器 ↔ Worker: v1 同进程（可信）；v2 网络层需要 mTLS
- Worker ↔ Tool: 可信
- Tool ↔ 系统: 受 allowlist 约束

**v1 安全机制**:
- **Subprocess argv 注入防御**: 工具必须用结构化参数构建 argv（用 `subprocess.run([...], check=True)`），**绝不**用 `shell=True` 或字符串拼接
- **文件系统**: 任何触文件的工具必须用 `SafePath` 辅助函数，路径在 `filesystem_allowlist` 内才放行
- **网络**: 任何触网络的工具必须用 `SafeURL` 辅助函数，URL 在 `network_allowlist` 内才放行；默认禁止 localhost、169.254/16、10/8、172.16/12、192.168/16（防 SSRF）
- **FFmpeg 二进制发现**: `shutil.which("ffmpeg")`，配置文件可覆盖；找不到就大声报错

**v1 不做**:
- AI 客户端和协调器之间的认证（stdio 模式假设本地用户）
- 工具执行的沙箱隔离（依赖 OS 用户权限）
- 协调器和 worker 之间的能力鉴权（v2 才需要）

### Error Model

**错误信封（兼容 JSON-RPC 2.0）**:
```json
{
  "error": {
    "code": -32001,
    "message": "人类可读的描述",
    "data": {
      "tool": "media.transcode",
      "args": {...},
      "stderr": "...",
      "traceback": "..."  // debug 模式才有
    }
  }
}
```

**错误码**:
- `-32001`: tool not found
- `-32002`: tool execution timeout
- `-32003`: tool raised an exception
- `-32004`: tool returned invalid output (非 JSON 可序列化)
- `-32005`: security violation (path/URL 不在 allowlist)

**AI 看到**: MCP server 返回 JSON-RPC error，客户端把错误转成对话里的 tool error。AI 可以读 `data.stderr` 来调试。

### Configuration

**配置文件**: `~/.evermcp/config.toml`

```toml
[general]
log_level = "INFO"  # DEBUG, INFO, WARNING, ERROR
log_file = "~/.evermcp/evermcp.log"

[ffmpeg]
binary = "ffmpeg"  # 或绝对路径
default_timeout_s = 600

[security]
filesystem_allowlist = ["~/data", "~/Downloads"]
network_allowlist = ["github.com", "pypi.org"]
denied_paths = ["~/.ssh", "~/.aws", "~/.config/gh"]
```

**优先级**: CLI flags > env vars (`EVERMCP_*`) > config file > defaults。

### Logging

**格式**: 结构化 JSON 输出到 stderr（不污染 stdio MCP 协议），人类可读格式输出到 log file。

**stderr 示例**:
```json
{"ts": "2026-06-24T15:00:00Z", "level": "INFO", "msg": "tool registered", "tool": "ffmpeg.transcode"}
```

**log file 示例**:
```
2026-06-24 15:00:00 INFO  tool registered tool=ffmpeg.transcode
```

FFmpeg 转码时，stdout 通过 `-progress pipe:1` flag 拿到机器可读进度（`out_time_ms` / `progress` / `speed` 等 `key=value\n` 行），每 1 秒聚合一次，写入 log file。AI 客户端看不到进度（stdio 限制）。

### Long-Running Jobs

**v1 姿态: 同步执行 + 超时 + `-progress` flag**

- AI 调用 `ffmpeg.transcode(...)` 后**同步等待**响应
- Worker 用 `subprocess.Popen` spawn FFmpeg，命令行加 `-progress pipe:1`（FFmpeg 官方机器可读进度接口），stdout 解析 `key=value\n` 格式
- 进度行（`out_time_ms`, `progress`, `speed`）每 1 秒聚合一次，写到 log file
- 如果 FFmpeg 超过 `ffmpeg.default_timeout_s`（默认 600s = 10min），子进程被 SIGTERM kill，返回 timeout 错误（code -32002）
- AI **看不到进度事件**，只能等（stdio MCP 限制）。一个 4K 转码可能要等 5-10 分钟，期间 Claude Desktop 看起来像"卡住"
- **partial output**: 超时时不返回部分输出（FFmpeg 输出文件可能损坏），明确告诉 AI "超时，已 kill"

**v2 改进**（不实现）: 异步 job + SSE 进度推送，让 AI 看到进度。

### Testing Strategy

**测试分层**:
1. **Unit tests** (`tests/unit/`): 每个工具函数独立测试，subprocess 用 mock
2. **Worker protocol tests** (`tests/worker/`): LocalWorker round-trip，错误信封形状
3. **Tool registry tests** (`tests/registry/`): hot-reload 模拟，文件 watcher 行为
4. **End-to-end tests** (`tests/e2e/`): mock MCP 客户端 → 协调器 → worker → 工具，真实 FFmpeg 用 fixture 文件调用
5. **Integration smoke test** (`tests/integration/`): 真的启动 MCP server，stdio 连接，list tools，call one，验证结果

**CI**: GitHub Actions，pytest + ruff + mypy，所有分层跑在 PR 上。

## Success Criteria

**v1.0 完成定义**:
1. AI 助手（Claude Desktop）能调用本机 FFmpeg 完成一次视频转码，端到端
2. AI 能调用 `io.read_file` 读 `~/data/` 下的文件
3. AI 能调用 `http.get` 访问 `github.com` 拿到响应
4. `tools/` 目录新增一个 `.py` 文件后，**新工具在 5 秒内出现在 worker 的 `list_tools()` 输出里**（用集成测试 client 验证，不需要走 AI 客户端）
5. Claude Desktop 配置文件示例可直接复制使用
6. README 完整：架构图、运行步骤、"加一个新工具"教程

**重要限制（stdio MCP 下的事实）**:
- 多数 stdio MCP 客户端（包括 Claude Desktop 当前版本）在握手时缓存工具列表
- 也就是说：**新工具 hot-reload 后，AI 要看到它需要重启客户端**（不是重启 server）
- 这是 stdio 协议的固有限制，不是 EverMCP 的 bug
- 验证手段：用集成测试 client 调 `list_tools()` 确认 worker 端已注册；然后重启 Claude Desktop，AI 就能调用

**v1.0 验证手段**:
1. 录一个 30 秒 demo 视频：让 Claude 转码一个 4K 视频到 1080p
2. 跑通 `tests/integration/`，所有 5 个分层通过
3. 自己用 1 周做真实任务（例如：批量转码某个文件夹、批量下载 README），记录：
   - 哪个工具好用、哪个难用
   - 哪个工具缺失（需要新加）
   - 错误信息够不够清晰

## Open Questions

下面这些**不是 v1 决策**，是 v2 才开始讨论的：

- **NPU 设备支持**: 需要 device capability 描述符（"我有 NPU"、"我能跑 llama.cpp 7B"），v1 worker 的 `get_capabilities()` 留接口但不实现
- **跨设备文件传输**: v1 假设 worker 只处理本地文件；v2 需要协调器层负责文件传输（S3 兼容 或 HTTP）
- **协调器-Worker 鉴权**: v2 加 mTLS
- **MCP transport 升级**: v2 加 SSE，让 hot-reload 真正对 AI 可见
- **Agent 自进化**（v3+ 愿景）: 架构里有 hook（`tools/` hot-reload 任意符合规范的脚本自动注册），但 v1 不做"Agent 自己写"的逻辑

## Reviewer Concerns

下面是 reviewer 提过但本版本没完全解决的细节问题。实现时遇到再处理：

- **Hot-reload 边界情况**: 工具文件被部分写入（编辑器原子保存前的瞬间）、in-flight 工具被新版本覆盖、工具文件被删除 — watchdog 行为需要实现时实测
- **SafeURL 子域名**: `network_allowlist = ["github.com"]` 不会匹配 `api.github.com`。v1 实现考虑自动允许子域名（带 warning 日志），或文档明确这点
- **`ToolContext` 取消机制**: v1 不做（YAGNI，stdio MCP 下 AI 客户端没有取消协议）
- **Watchdog 平台差异**: macOS 用 FSEvents，Linux 用 inotify，Windows 用 ReadDirectoryChangesW。某些编辑器（vim/某些 IDE）rename-then-write 模式可能漏事件。v1 接受这个风险
- **FFmpeg `-progress` 解析**: 不同 FFmpeg 版本字段名可能不同（`out_time_ms` vs `out_time_us`）。v1 测试 6.x/7.x 两个版本就够
- **非 JSON 可序列化返回值**: 工具返回 `datetime`/`Path`/`Decimal` 时，worker 用 `json.dumps(default=str)` 兜底，文档里说明
